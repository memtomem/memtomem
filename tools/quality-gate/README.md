# Replay quality gate — CI fixture + driver

This directory holds the committed fixture assets and the CI driver for the
policy-driven **replay regression gate** (issue #1833, Quality Lab Q4). The gate
replays a small, deterministic set of evaluation cases against a fixed retrieval
profile and fails when the candidate ranking drifts from a committed baseline
beyond what the policy allows.

The corpus is the same synthetic, public `corpus_v2` fixture used by
`tools/retrieval-eval/` — no customer records, user memories, or private
documents. See its
[provenance and privacy contract](../../packages/memtomem/tests/fixtures/corpus_v2/README.md).

## What is committed here

| File | Author | Purpose |
|------|--------|---------|
| `fixture.json` | hand-written | Manifest: corpus dir, pinned `as_of`, BM25-only config overrides, and 12 `case_defs` whose relevance is declared as `relevant_globs` over corpus files. |
| `cases.json` | **generated** | Portable `eval_case_set` envelope — the `relevant_globs` resolved to chunk `content_hash` labels. Imported at replay time. |
| `baseline_replay.json` | **generated** | Canonical baseline replay report the gate compares against. |
| `policy.json` | hand-written | The `GatePolicy` — the single source of every threshold (no thresholds live in the workflow YAML). |
| `run_gate.py` | hand-written | The CI driver (check + refresh modes). |

`cases.json` and `baseline_replay.json` are regenerated by `run_gate.py`, never
edited by hand.

## Determinism

- **BM25 only** (`embedding.provider=none`, `search.enable_dense=false`) — no
  model is downloaded and no dense/rerank stage runs, so replay is deterministic.
- **`as_of` is pinned** far in the future (`1893456000` = 2030-01-01) so temporal
  validity and time-decay are stable.
- **Labels key on `content_hash`** (`sha256(NFC(content))`), and the corpus ships
  with `* text=auto eol=lf`, so labels resolve identically across machines.

The driver copies the corpus into a temp directory before indexing (indexing
writes `.<name>.lock` sidecars next to source files) and scrubs every
`MEMTOMEM_*` variable behind a throwaway `$HOME`, so a run never touches your
real `~/.memtomem/`.

## Expected-false compatibility flags

The comparison's `compatibility` block reports `corpus_match` and `index_match`
as **false** for a committed baseline vs a freshly indexed candidate, and this is
expected — both fingerprints fold wall-clock `created_at`/`updated_at` timestamps
and machine-hashed absolute paths, so they can never match across two indexing
runs. `policy.json` therefore requires only `case_set_match`, `profile_match`,
`as_of_match`, and `deterministic_inputs`. For the same reason, regenerating the
baseline produces a diff in the `fingerprints.corpus` / `fingerprints.index`
fields even when nothing about the ranking changed.

## Complete labels and the precision floor

Each case's labels are **complete over the corpus**: the `relevant_globs` files
are labelled relevant and every other corpus chunk is labelled `not_relevant`
(this is why `cases.json` is large — it is generated, not hand-edited). Complete
judgments keep precision *comparable* for every case, which lets the policy carry
a `precision` floor of `0.0`. That floor closes a gap a relevant-only fixture
would leave open: a candidate that appends an off-target result after the rank-1
hit leaves the four rank metrics unchanged, but because the appended chunk is a
labelled `not_relevant` hit it lowers precision, so the gate catches it instead
of classifying the case as `unchanged`.

**Precision-cohort coverage.** The floor only sees the *comparable* subset, so a
compared case whose precision goes incomplete is silently excluded from it. If a
corpus change makes a case retrieve a *new* unlabelled chunk, that case drops out
of the precision cohort; with the other cases' precision flat the mean delta
stays at zero, and because the gate does not require `corpus_match` (its
fingerprint folds timestamps) the case would otherwise slip through as
`unchanged`. `policy.json` closes this with `max_incomplete_precision_cases: 0`:
any compared case with incomplete precision is a `precision_cohort_coverage`
violation rather than a silent exclusion (#1837). A corpus edit that legitimately
changes retrieval therefore still requires refreshing the baseline in the same PR
— now enforced, not just advised.

## Run it locally

```bash
# Check mode (what CI runs): exit 0 pass / 1 policy violation / 2 infra error.
uv run python tools/quality-gate/run_gate.py; echo "exit=$?"
```

## Refresh the baseline

Regenerate `cases.json` + `baseline_replay.json` whenever you make an **intended**
ranking or chunking change (or add/adjust cases in `fixture.json`), then review
and commit the diff in the same PR:

```bash
uv run python tools/quality-gate/run_gate.py --refresh-baseline
git diff tools/quality-gate/cases.json tools/quality-gate/baseline_replay.json
```

The refresh self-checks that the fresh baseline passes its own policy before
writing.

### Canonical producer: Ubuntu

BM25 ranking can vary slightly across OS / SQLite builds, so the **canonical
baseline is produced on Ubuntu** (the CI runner). A maintainer on another OS
should not hand-commit a locally produced baseline: run the CI workflow with
`workflow_dispatch` and input `refresh_baseline: true`, download the uploaded
`cases.json` + `baseline_replay.json` (+ `versions.txt` recording the Python and
SQLite versions), and commit those. A locally produced (e.g. macOS) baseline can
diverge in raw BM25 scores from the Ubuntu runner; the gate thresholds on
rankings and metrics, not raw scores, so this does not break the check, but the
committed baseline should still come from the canonical producer.

## CI

The `quality-gate` job in `.github/workflows/ci.yml` runs check mode on every push
and PR as a **required** check: a policy violation fails the PR. The
`workflow_dispatch` path additionally supports `refresh_baseline: true` to
regenerate and upload the canonical assets. Any intended ranking change must
refresh the baseline in the same PR, or the gate will (correctly) fail.
